# SCAPES Consortium: FAIR Open-Science Pipeline

## 1. Architectural Overview
To manage the operational tension between strict data privacy (handling identifiable human GPS tracks and serology) and funder mandates for open, FAIR (Findable, Accessible, Interoperable, and Reusable) code and data, the SCAPES consortium implements a **Dual-Repository Architecture**.

All primary data ingestion, cleaning, and preliminary modeling occur within a secure, access-controlled **Private Monorepo**. At manuscript submission and publication milestones, automated continuous integration pipelines sanitize, decouple, and push code and anonymized data to topic-specific **Public Release Repositories** under the RISK-Lab GitHub organization.

## 2. The Private-to-Public Pipeline    

```text
[Field Data Collection] 
   │ (Raw GPS Tracks, ODK Questionnaires, RT-PCR/ELISA Results)
   ▼
[Private Consortium Repository] <─── (Restricted Access: PSU / Tulane / NHM)
   │ 
   ├─► 1. Data Sanitisation Script (Applies Affine Null-Space Transformation)
   ├─► 2. Code Scraping (Removes absolute local file paths & sensitive API keys)
   ├─► 3. Unit Testing & Quarto Render Verification
   │
   ▼ 
[Public Release Repository] <───── (Open Access: e.g., RiskLabPSU/landscapes)
   │
   ├─► Automated DOI via Zenodo Integration
   ▼
[Permanent Open Access Archive] ───► Referenced in Master Dictionary & Publication
```

## 3. Automated Sanitation & Release Protocols
When an analytical stream (e.g., Aim 1D Rodent $N$-mixture modeling) reaches publication maturity, the release pipeline executes three mandatory automated steps:

### A. Identifier & Path Stripping
* An automated linting script scans all R, Python, and Quarto (`.qmd`) scripts to strip absolute local file paths (e.g., `C:/Users/...` or `/home/server/...`), replacing them with relative project-root paths using the R `here` package or Python `pathlib`.
* Environment variables, API tokens, and internal server IP addresses are purged and replaced with `.env.example` templates.

### B. Data Ingestion Decoupling
* Scripts reading directly from secure consortium SQL databases or restricted AWS S3 buckets are decoupled. 
* The ingestion block is redirected to read from the sanitized, anonymized CSV/RDS analytical products generated by the `data_governance.md` protocol, which are bundled into the public repository's `/data/` directory or hosted on an open data repository (e.g., Dryad/Zenodo).

### C. Persistent Identifier (DOI) Minting
* Every public release repository is natively integrated with **Zenodo** via a `.zenodo.json` metadata file and a `CITATION.cff` file.
* Upon triggering a formal GitHub Release (e.g., `v1.0.0 - Publication Release`), Zenodo automatically archives a frozen snapshot of the codebase and mints an immutable Digital Object Identifier (DOI).
* This DOI is immediately appended to the **Master Repository Index** in the primary `SCAPES-protocol` repository.

## 4. Reusability and Modular Containerization
To ensure that external researchers can execute SCAPES models without dependency or environment conflicts:
* **Package Versioning:** All R-based modeling pipelines (e.g., `spOccupancy`, `spAbundance`, `cmdstanr`, `brms`, `tidyterra`) are governed by `renv` lockfiles (`renv.lock`), locking exact CRAN and GitHub package versions.
* **Spatial Libraries:** Core geospatial dependencies (`GDAL`, `PROJ`, `GEOS`) required for UAV orthomosaic processing and spatial modeling are documented in Dockerfiles included within each public repository, enabling one-click containerized replication.
